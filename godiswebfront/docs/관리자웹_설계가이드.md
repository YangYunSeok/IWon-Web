## GODIS Web 문서 구성 및 정리 가이드

이 문서는 `godiswebfront/docs` 폴더 내의 설계 문서를 **개념 설계**, **논리 설계**, **물리 설계** 단계별로 분류하고 각 문서의 주요 내용을 정리한 가이드입니다.

#### 설계 단계별 개요

| 구분 | 설계 단계 | 주요 문서 | 핵심 내용 정리 |
| -- | ----- | -------- | -------- |
| 1  | 개념설계  | 기본설계문서(레거시 참고) | 관리자 시스템의 전체 범위/업무 흐름을 정의한 참고 문서(SSOT 아님) |
| 2  | 논리설계  | design(SSOT) | 관리자 UI 구현을 위한 API 계약(@codegen) 및 데이터 모델/공통 타입/에러, 화면별 UX/플로우를 단일 기준으로 관리 |
| 3  | 물리설계  | 화면 표준 개발 가이드, DB 테이블 설계서 | 실제 구현 단계의 기준 문서(UI 표준, DB 물리 구조, 구현 규칙) |

### 1. 개념 설계 (Conceptual Design)

**개념 설계**는 시스템이 *무엇을* 하는지에 초점을 맞추며, 사용자의 요구사항, 전체 시스템 구조, 고수준의 사용자 인터랙션 및 외관을 정의합니다.

| 문서명 | 문서분류 | 내용 |
| --- | --- | --- |
| 웹UI및설계가이드(관리자용).md | 개념설계 | 관리자 웹 UI/UX에 대한 레거시 가이드(참고용). SSOT와 충돌하면 SSOT 우선 |
| api_및_데이터모델_명세서(관리자용).md | 개념설계 | 레거시 API/모델 명세(참고용). SSOT(@codegen 및 model 문서) 기준으로 갱신됨 |

#### 문서 목록 및 내용

*   **대체 참고(SSOT):**
    *   **SSOT 허브/우선순위:** `docs/design/_index.md`
    *   **화면 목록/매핑:** `docs/design/ui/_screen-map.md`
    *   **화면별 UI 설계(SSOT):** `docs/design/ui/admin/*`
    *   **API 계약(@codegen):** `docs/design/api/web-admin.md`
    *   **모델/타입/에러:** `docs/design/model/*`

*   **`기본설계문서/웹UI및설계가이드(관리자용).md`**
    *   **내용:** IWon Admin Web 시스템의 UI/UX 디자인을 상세히 설명하는 가이드입니다. 이전 문서인 "목록 정리"를 기반으로 각 화면에 대한 더 시각적이고 인터랙티브한 디자인 사양을 제공합니다. 내비게이션 구조를 설명하고 로그인, 대시보드, 지갑 관리, 코인 분배, 거래 내역 조회, 승인 관리, 월별 계획 등 주요 화면 각각에 대한 상세 레이아웃 및 컴포넌트 사양을 제공합니다. 각 화면에 대해 KPI 표시, 목록을 위한 테이블 컬럼, 액션 버튼 등 핵심 정보 및 액션을 설명합니다. 또한 API 디자인 문서를 참조하고 공통 오류 코드에 대한 UI 레벨 처리를 지정합니다.
 
*   **`기본설계문서/api_및_데이터모델_명세서(관리자용).md`**
    *   **내용:** I-Won Admin 시스템의 API 및 데이터 모델을 상세히 기술하는 명세서입니다. 인증(`Bearer Token` 및 `operator`, `approver` 역할), 데이터 타입(`IWC`, `ISODateString`), 오류 코드와 같은 공통 규약을 정의합니다. `SupplySummary`, `EmployeeWallet`, `AdminTransaction`, `MonthlyPayee`와 같은 주요 엔티티에 대한 데이터 모델을 제공합니다. 또한 대시보드, 지갑 관리, 코인 분배, 승인 등 각 관리 기능에 대한 REST API 엔드포인트(HTTP 메소드, 경로, 인증 요구사항, 요청/응답 DTO)를 명시합니다. 이 사양은 코드 생성 도구와 함께 사용되도록 설계되었습니다.

#### 문서 목록 및 내용

*   **`design/_index.md`**
    *   **내용:** 설계 문서의 중앙 허브 또는 인덱스 역할을 하며, 다양한 설계 문서의 구조와 우선순위를 설명합니다. 이 파일은 문서들이 물리적으로 분리되어 있지만 하나의 논리적인 SSOT(Single Source Of Truth)를 형성함을 명확히 합니다. 충돌 해결을 위한 우선순위를 설정합니다: API 계약(`design/api/`)이 가장 높은 권한을 가지며, 데이터 모델(`design/model/`), 그 다음 UI/UX 동작(`design/ui/`) 순입니다. 또한 API, 모델, UI 설계를 위한 주요 문서를 나열하고, 특히 `@codegen` 블록에 대한 문서 검증 및 린팅 지침을 포함합니다.

*   **`design/api/web-admin.md`**
    *   **내용:** Admin Web API 계약의 SSOT(Single Source Of Truth) 역할을 합니다. 관리자 콘솔을 위한 API 사양을 정의하며, 통신 프로토콜, 인증 규칙(`Bearer Token`과 `operator` 및 `approver` 역할), 그리고 다른 `design/model/` 파일에 있는 공통 데이터 모델 및 오류 코드 참조를 포함합니다. 이 문서는 대시보드(`admin/stats/*`), 지갑 관리(`admin/employees`), 코인 분배, 승인, 월별 지급 대상자 관리(`admin/monthly-payees`)와 같은 리소스에 대한 상세 엔드포인트 사양을 제공합니다. 각 엔드포인트는 ID, 메소드, 경로 및 요청/응답 타입을 지정하는 `@codegen` 블록으로 정의됩니다.

*   **`design/model/web-admin.md`**
    *   **내용:** 웹 관리자 데이터 모델의 SSOT(Single Source Of Truth) 역할을 합니다. 오래된 통합 설계 문서에서 구조화된 `design/model/` 디렉토리로 마이그레이션되었습니다. API 요청 및 응답에 사용되는 특정 데이터 구조(DTO)를 정의합니다. 주요 모델에는 대시보드를 위한 `SupplySummary` 및 `DailyMetrics`; 월별 지급 계획을 위한 `MonthlyPayee` 및 관련 대량 작업 DTO; 직원 및 지갑 정보를 위한 `EmployeeWallet`; 그리고 거래 및 승인 워크플로우를 위한 `AdminTransaction` 및 `ApprovalRequest`가 포함됩니다. 각 모델에 대한 필드 이름, 타입 및 설명을 제공하여 API에 대한 일관된 데이터 계약을 보장합니다.

*   **`design/model/web-common-types.md`**
    *   **내용:** 관리자 웹 애플리케이션 전반에 걸쳐 사용되는 공통 데이터 타입 및 열거형에 대한 SSOT(Single Source Of Truth) 역할을 합니다. 오래된 문서에 흩어져 있던 타입 정의를 단일의 구조화된 파일로 통합합니다. `IWC`(코인 값의 정수 단위)와 같은 스칼라 타입과 핵심 비즈니스 로직 상태를 나타내는 다양한 문자열 리터럴 타입(열거형)을 정의합니다. 이러한 열거형에는 `UserRole`(`'operator'` | `'approver'`), `WalletStatus`(`'uncreated'` | `'active'` | `'frozen'`), `TxType`(`'mint'` | `'burn'` | `'transfer'`), `ApprovalStatus`(`'pending'` | `'approved'` | `'rejected'`), `CoinUsageCategory`(`'welfare'` | `'payment'`)가 포함됩니다.

*   **`design/model/web-error-codes.md`**
    *   **내용:** 웹 관리자 콘솔에 특화된 오류 코드에 대한 SSOT(Single Source Of Truth) 역할을 합니다. 고유한 오류 코드, 설명 및 각 코드에 대한 권장 UI/UX 처리 전략을 정의합니다. 예를 들어, `DISCREPANCY_DETECTED`는 대시보드에 빨간색 경고를 트리거하고, `INSUFFICIENT_RESERVE`는 승인 버튼을 비활성화합니다. 또한 오류 모델 재사용 규칙과 새로운 관리자 관련 오류가 도입될 때 특정 업데이트 순서(이 문서 -> API 문서 -> UI 문서)를 명시합니다.

*   **`design/ui/_screen-map.md`**
    *   **내용:** 웹 관리자 애플리케이션을 위한 화면 매핑 테이블로, UI 설계 문서, 구현된 기능 및 백엔드 API를 연결하는 중앙 참조 역할을 합니다. 각 화면(`화면키`)을 이름, 파일명, 메뉴 키, 라우트, 권한 키(TBD), 해당 UI 문서, 관련 API 리소스 및 소스 경로(TBD)에 매핑합니다. SSOT 우선순위(API 계약 > UI 동작)를 재확인하며, 이 문서가 계약을 정의하기보다 *연결/매핑*을 위한 것임을 강조합니다.

*   **`design/ui/admin/web-admin-console.md`**
    *   **내용:** 웹 관리자 콘솔 및 로그인 프로세스에 대한 개요를 제공하며, 기존 UI 가이드에서 수정된 내용입니다. 사이드바 메뉴 항목(대시보드, 지갑 관리, 코인 분배, 거래 내역 조회, 승인 관리, 월별 계획)을 포함한 내비게이션 구조를 상세히 설명합니다. 로그인 화면 UI 컴포넌트(제목, 로고, ID/비밀번호 필드, 로그인 버튼) 및 오류 처리를 설명하며, 2단계 인증은 보류 중임을 명시합니다.

*   **`design/ui/admin/web-approval.md`**
    *   **내용:** 웹 관리자 승인 관리 화면의 UI/UX 사양을 상세히 설명합니다. Mint/Burn 제안에 대한 사전 통제 및 최종 실행을 목적으로 하며, 주로 "Approver" 역할에 사용되고 "Operator"는 보기 전용 접근 권한을 가집니다. 레이아웃에는 KPI 배너(대기 승인, 경고, 예상 준비금 비율 표시), 대기 승인 목록 테이블(유형, 제목, 금액, 시스템 사전 검증 결과, 요청 시간), "상세 보기" 액션이 포함됩니다. "승인 상세 팝업"에는 전후 유통 비교, 문제 대상 목록, Excel 다운로드 옵션이 제공됩니다.

*   **`design/ui/admin/web-coin-dist.md`**
    *   **내용:** 웹 관리자 코인 분배(CoinDist) 화면의 UI/UX 사양을 설명합니다. 복지/결제 코인의 발행(Mint), 회수(Burn), 대량 업로드 작업을 처리하는 것이 주 목적입니다. 운영자가 제안을 시작하고 승인자가 승인합니다. 화면 레이아웃은 상단 정리(선택 인원 수, 총 금액, 현재 사용자 역할), 왼쪽 대상 선택 영역(조직 트리, 검색, 다중 선택 칩), Mint, Burn, 대량 업로드 탭이 있는 오른쪽 제어판으로 구성됩니다.

*   **`design/ui/admin/web-dashboard.md`**
    *   **내용:** 웹 관리자 대시보드 화면의 UI/UX 사양을 상세히 설명합니다. 재무 건전성, 발행 통제, 온체인 처리 상태를 실시간으로 모니터링하고 불일치에 대한 즉각적인 경고를 제공하는 것이 목적입니다. 로그인 또는 '대시보드' 사이드바 항목 클릭 시 화면이 트리거되며, `GET /admin/stats/supply` 및 `GET /admin/stats/daily`를 동시에 호출합니다. UI는 여러 섹션으로 나뉩니다.

*   **`design/ui/admin/web-wallet-mgmt.md`**
    *   **내용:** 임직원 지갑 생성 여부/상태/Explorer 링크를 조회하고, 선택 사번 대상으로 지갑 생성 플로우(`GET /admin/employees`, `POST /admin/wallets/create`)를 정의합니다.

*   **`design/ui/admin/web-financial-closing.md`**
    *   **내용:** 재무회계결산 화면의 탭 구성(분개/보고서), 기간 필터, 다운로드 규칙을 정의합니다. 단, API 계약은 `docs/design/api/web-admin.md`에서 아직 TBD로 남아있습니다.

*   **`design/ui/admin/web-monthly-plan.md`**
    *   **내용:** 웹 관리자 월별 계획 화면의 UI/UX 사양을 설명합니다. 월별 복지 코인 분배 계획을 사전 설정하고 확정하는 것이 목적입니다. 주요 기능에는 이전 달 데이터 재사용, 지급 대상자 정보 편집, 확정 프로세스가 포함됩니다. 화면 상단에는 참조 월 선택(`YYYY-MM`) 및 계획 상태(초안/확정) 표시 섹션이 있습니다. 본문에는 지급 대상자 세부 정보(사원 ID, 이름, 금액, 비고)가 포함된 테이블이 표시됩니다.

*   **`design/ui/admin/web-tx-history.md`**
    *   **내용:** 웹 관리자 거래 내역(TxHistory) 화면의 UI/UX 사양을 설명합니다. 감사 관점에서 모든 온체인 및 오프체인 거래를 추적하고, 검색, 필터링, 상세 미리보기 기능을 제공하는 것이 목적입니다. 상단 섹션에는 날짜 범위(기본값 30일), 거래 유형(발행, 소각, 전송), 코인 유형(복지, 결제) 및 통합 검색 필드(사번, 이름, TxHash)가 있습니다.

PostgreSQL 데이터베이스에 맞춰 설계되었습니다. 관리자 사용자(`Admin Users`, `Admin Roles`), 감사(`Audit Logs`), 직원 정보(`Employees`), 지갑 상세 정보(`Wallets`, `Coin Types`), 거래(`Wallet Transactions`)를 관리하기 위한 테이블을 정의합니다. 또한 승인 워크플로우(`Approvals`, `Approval Targets`) 및 해당 승인 실행(`Admin Transactions`)을 위한 테이블을 포함합니다. 대시보드를 위해서는 일별 통계(`Stats Daily`) 및 과거 공급 스냅샷을 위한 테이블을 제안합니다. 월별 지급 대상자(`Monthly Payees`) 관리 테이블도 정의합니다.
